Required Fields

The Learning Aim Subjection has some enumerated fields which if not filled in accurately stop the user from inserting or deleting elements or saving the document. However if left blank then saving can still occur.


Some fields have enumeration or pattern matching requirements in the schema which are recognized by Xopus. Often these fields must be filled in correctly before continuing to edit with full insert and delete capabilities or being able to save the document.


We have removed xml schema Criterion ID pattern matching in the units that were causing such trouble to save or insert further learning objects before having entered any Criterion ID data and rather now highlight the Criterion ID using java script with a dotted line which stays until the entered data is correct. This currently goes away however when interacting with xopus in other ways such as inserting an element.

Probably a toolbar button could be created that calls a javascript function to check for any such remaining errors.

GSCE English/Math

Currently GSCE English/Math are enumeration values of Yes or No so if you enter something other than this then it needs to be corrected before continuing. However you can continue to save, edit, insert, and delete when the value has not been edited so this should be a good balance between enforcing accuracy and being more free to edit.



Sections

Numbering

Because each section is being authored in a separate locations. The section number does not appear as there is no easy way to determine what number. Each section has a section numbering boolean attribute which is adjustable in the properties slider:

  1. Section1 the section numbering is true by default

  2. Section2 -Section6 the section numbering is false by default

  3. The XSL to generate the final HTML would be able to figure our what the numbering should be based on these values and the order of the sections when combined.

  4. This functionality could be extend to include a style attribute if letters were desired.

Lists with header and continued numbering

  1. Pg 303 in the pdf

  2. We have not implemented list with headers and continued numbering. It's not default xopus list behavior. We would have to modify the default xopus list behaviour to some extent. Should we allocate time developing that?
