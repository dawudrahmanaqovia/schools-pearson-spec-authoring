Quirky Editing Features

Unit

Units being the most complicated seem thus far to have some of the most restrictions in editing.

Learning Aim Subsections

The Learning Aim Subjection has some required fields which if not filled in accurately stop the user from inserting or deleting elements or saving the document. This restriction is difficult to understand and we have submitted feedback with Xopus requesting information of whether the system can be adjusted to allow required fields to not have to be all filled out and still allow insertion and deletion. Perhaps it does make sense however that these fields should be filled in before saving in order for the xml to be valid.


So generally a user would probably start editing currently by filling in the required fields and then have freedom to edit and save as desired.


We could reduce which initial elements are inserted in the xopus configuration files so that for example no learning assessment criteria was inserted by default so that these required fields are not all loaded at the same time. We initially make the default config as it was so it was clear to the user what was available to edit.


Required Fields

Some fields have enumeration or pattern matching requirements in the schema which are recognized by Xopus. Often these fields must be filled in correctly before continuing to edit with full insert and delete capabilities or being able to save the document.

GSCE English/Math

Currently GSCE English/Math are enumeration values of Yes or No

Criterion ID

Currently the Criterion id follows the following regex.


If you enter a Criterion ID which is not accurate Xopus will not let you do insert, delete or save the document until it's changed.


\s*\(([0-9][A-Z]\.(P|M|D)?[0-9][0-9]?|,|\s)+\)|([0-9][A-Z]\.(P|M|D)?[0-9][0-9]?)\s*


Which specifies the possible styles of criterion ID similar to the following:

1X.1

1X.P1

1A.M1

1X.13

(1A.M1)

(1Z.1)

(1A.M1, 1A.2, 2A.1)

( 1A.M1 , 1A.2, 2A.1 )
